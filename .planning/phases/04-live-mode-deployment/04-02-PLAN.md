<objective>
Package the screener as a production-grade `node:22-alpine` Docker image: multi-stage build (builder + runtime), non-root user, exec-form ENTRYPOINT for clean PID-1 signal handling, and a `.dockerignore` that prevents host artifacts (node_modules, secrets, build output, planning docs) from leaking into the image. Purpose: INFRA-01 requires the application to run as a short-lived `node:22-alpine` container that exits with code 0/1 and accepts config via volume mount and secrets via `--env-file`. The container is the unit of deployment — without it, no cron entry can run.
Output: `Dockerfile` and `.dockerignore` at the project root. No application code changes — this plan is intentionally orthogonal to Plan 1, allowing parallel execution.

<!-- Build-time and runtime contracts. Use these directly. -->

From package.json:
```json
{
  "type": "module",
  "main": "dist/index.js",
  "scripts": {
    "build": "tsc"
  },
  "engines": { "node": ">=22.0.0" }
}
```
The image entrypoint must be `node dist/index.js` because `package.json#main` points there and `npm run build` (= `tsc`) produces `dist/`.

From tsconfig.json: `outDir: "dist"`, `rootDir: "src"` — so `dist/` is the only build output directory.

From the existing `src/index.ts`:
- Reads `process.env['CONFIG_PATH']` (defaults to `./config.yaml`) → so the runtime expects either `/app/config.yaml` (default working dir is `/app`) OR an absolute path supplied via env.
- Reads `BAMBOOHR_API_KEY`, `BAMBOOHR_SUBDOMAIN`, `OPENAI_API_KEY`, `LIVE_MODE` from env.
- `import 'dotenv/config'` is the first import — but inside a Docker container with `--env-file`, dotenv finds no `.env` file and is a no-op. That's correct: `--env-file` injects vars at the container level, not via a file inside the container.

Files in the host project root that the build stage needs:
- `package.json`, `package-lock.json` — for `npm ci`
- `tsconfig.json` — for `tsc`
- `src/` — TypeScript source

Files that MUST NOT enter the image:
- `node_modules/` (host macOS binaries are ABI-incompatible with Alpine)
- `.env`, `.env.*` except `.env.example` (secrets)
- `dist/` (rebuilt inside the container; stale host dist is misleading)
- `.git/`, `.planning/`, `*.md` (no runtime value)
- `Dockerfile`, `.dockerignore` (no runtime value)
</interfaces>

    Create a new file `Dockerfile` at the project root (alongside `package.json`). Write exactly this content:

    ```dockerfile
    # syntax=docker/dockerfile:1
    # BambooHR Candidate Screener — production image (INFRA-01)
    # Multi-stage build: stage 1 compiles TypeScript with devDependencies;
    # stage 2 ships only dist/ and runtime deps on node:22-alpine.

    # ──────────────────────────────────────────────────────────────────────────────
    # Stage 1: build (has devDependencies for tsc)
    # ──────────────────────────────────────────────────────────────────────────────
    FROM node:22-alpine AS build
    WORKDIR /app

    # Copy package files first so npm ci layer caches when source changes but deps don't.
    COPY package.json package-lock.json ./
    RUN npm ci

    # Copy source and tsconfig for the TypeScript build.
    COPY tsconfig.json ./
    COPY src ./src

    RUN npm run build

    # ──────────────────────────────────────────────────────────────────────────────
    # Stage 2: production (no devDependencies, no source .ts)
    # ──────────────────────────────────────────────────────────────────────────────
    FROM node:22-alpine AS production
    WORKDIR /app

    # Non-root user for runtime (Alpine uses BusyBox addgroup/adduser, not GNU).
    RUN addgroup -S screener && adduser -S screener -G screener

    # Copy compiled output and package manifests from the build stage.
    COPY --from=build /app/dist ./dist
    COPY --from=build /app/package.json ./
    COPY --from=build /app/package-lock.json ./

    # Production-only dependencies; leaves devDependencies (tsx, typescript, @types) out.
    RUN npm ci --omit=dev

    # The container reads CONFIG_PATH from env (defaults to ./config.yaml in src/index.ts).
    # Operators mount the config at /app/config.yaml via:
    #   docker run --rm --env-file /etc/screener.env \
    #     -v /path/to/config.yaml:/app/config.yaml:ro \
    #     bamboohr-screener:latest
    # Secrets (BAMBOOHR_API_KEY, BAMBOOHR_SUBDOMAIN, OPENAI_API_KEY, LIVE_MODE) come from
    # --env-file at runtime — never baked into the image (D-06).

    USER screener

    # Exec form ENTRYPOINT — process is PID 1, receives SIGTERM cleanly from `docker stop`.
    # No CMD: the container has a single execution path with no argument overrides.
    ENTRYPOINT ["node", "dist/index.js"]
    ```

    Do NOT add `EXPOSE`, `HEALTHCHECK`, `ENV`, or `CMD`. The container makes outbound HTTP only, has no inbound port, exits on completion (health = exit code 0/1), and has a single fixed entrypoint.

    Do NOT use `npm install` — use `npm ci` for reproducible installs from `package-lock.json` in both stages.

    Do NOT copy `node_modules/` from host — `.dockerignore` (Task 2) will exclude it, but the explicit `COPY package.json package-lock.json ./` followed by `RUN npm ci` is the canonical pattern.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Host filesystem → Build context | Files copied via `COPY` cross from host to image; `.dockerignore` is the gate |
| Image registry → Operator runtime | If the image is ever pushed to a registry, anything in image layers is exposed |
| Operator → Container | Runtime secrets injected via `--env-file`; config via `-v` volume mount |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-08 | Information Disclosure | Secrets baked into image layers via accidental `COPY .env` | mitigate | `.dockerignore` excludes `.env` and `.env.*` (with explicit `!.env.example` exception). Operator runs `docker history --no-trunc` in Task 3 to confirm no credential strings appear in layers. |
| T-04-09 | Tampering | Stale or incompatible host `node_modules/` copied into image | mitigate | `.dockerignore` excludes `node_modules/`. Both build stages run `npm ci` to install Alpine-compatible binaries inside Docker. |
| T-04-10 | Information Disclosure | Planning docs (`.planning/`) leaking internal architecture into a published image | mitigate | `.dockerignore` excludes `.planning/`, `.claude/`, `*.md`, `.git/`. |
| T-04-11 | Elevation of Privilege | Container running as root could exploit a kernel bug to escape namespace | mitigate | `RUN addgroup -S screener && adduser -S screener -G screener` followed by `USER screener` — process runs as unprivileged user. No file writes outside `/app` are needed. |
| T-04-12 | Denial of Service | `docker stop` blocked because process is wrapped by `/bin/sh -c` and never receives SIGTERM | mitigate | Exec-form `ENTRYPOINT ["node", "dist/index.js"]` ensures `node` runs as PID 1 and receives signals directly. No shell wrapper. |
| T-04-13 | Information Disclosure | `dotenv` accidentally loading a `.env` file from inside the container | accept | Inside the container at `/app`, no `.env` file exists (excluded from build context). `dotenv/config` is a no-op. Operators rely solely on `--env-file` for credential injection. |
| T-04-14 | Tampering | Operator launches the container with the wrong volume mount path, picking up an empty default config | mitigate | The existing `loadConfig()` function exits 1 with a clear error if `./config.yaml` is missing or invalid. The README in Plan 3 documents the exact `-v` mount syntax. |

</threat_model>
