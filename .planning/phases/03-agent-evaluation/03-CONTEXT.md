# Phase 3: Agent Evaluation - Context

**Gathered:** 2026-05-02
**Status:** Ready for planning

<domain>
## Phase Boundary

For each candidate that passed hard rules and has a successfully extracted CV, the system invokes a GPT-4o evaluation via the OpenAI Agents SDK. The agent receives `CandidateContext` (cvText, applicationAnswers, hardRuleResult) and evaluates each configured soft criterion (required and optional), producing a structured `EvaluationResult` with per-criterion pass/fail + rationale and a recruiter comment. Results are logged as `EvaluationResult` JSON to stdout. In dry-run mode, zero stage transitions and zero comments are written to BambooHR. Hard-rule fails, PDF failures, and errors continue to use the existing `CandidateDecision` log path.

</domain>

<decisions>
## Implementation Decisions

### Soft Rules YAML Shape

- **D-01:** Each soft rule entry has two fields: `label` (human-readable, used verbatim in log output and recruiter comment) and `description` (the evaluation criterion description sent to GPT-4o as the evaluation prompt for that criterion). Mirrors the hard-rule style from Phase 1.

- **D-02:** Soft rules are split into `required` and `optional` sections in config.yaml. `required` criteria are dealbreakers — ALL must pass for the candidate to receive a `pass` outcome. `optional` criteria are nice-to-haves — evaluated and included in output but never block the outcome.

- **D-03:** Pass logic is all-required-must-pass. No min-count threshold. If all required criteria pass → `pass`. If any required criterion fails → `fail`. GPT-4o response failure → `needsReview`.

- **D-04:** GPT-4o outputs pass/fail + a one-line rationale string per criterion. The rationale is grounded in the candidate's actual CV/answers (generated by GPT-4o, not the rule description verbatim).

- **D-05:** Optional criteria evaluations ARE included in the `EvaluationResult` output and in the recruiter comment (under a separate "Optional" section). Gives recruiters the full picture.

Example config shape for `softRules` (peer to `hardRules` in config.yaml):
```yaml
softRules:
  required:
    - label: "Strong technical experience"
      description: "Candidate demonstrates 3+ years of relevant engineering experience based on CV"
    - label: "Clear written communication"
      description: "Application answers are articulate, specific, and professional"
  optional:
    - label: "Open-source contributions"
      description: "Candidate has personal or open-source projects relevant to the role"
```

### Recruiter Comment Format

- **D-06:** Structured list format — fixed sections: outcome header, Met (required), Unmet (required, if any), Optional results, and a fixed footer `[Auto-screened by AI — final decision rests with recruiter]`. Scannable in seconds when reviewing many candidates.

- **D-07:** Comment covers soft evaluation only. Hard-rule pass is implicit — any candidate that reaches soft evaluation already cleared hard rules. No hard-rule summary in the comment.

- **D-08:** Per-criterion rationale in the comment is GPT-4o generated (grounded in actual candidate content), not the rule description from config.yaml.

Example comment (pass):
```
PASS — Soft Evaluation

Met (required):
• Strong technical experience: N years of relevant backend engineering per CV
• Clear written communication: Application answers are specific and show clear thinking

Optional (met):
• Open-source contributions: Active GitHub profile with 3 relevant repos linked in CV

[Auto-screened by AI — final decision rests with recruiter]
```

Example comment (fail):
```
FAIL — Soft Evaluation

Met (required):
• Clear written communication: Application answers are coherent

Unmet (required):
• Strong technical experience: CV shows 1 year experience, does not meet the 3+ year threshold

Optional (met):
• Open-source contributions: Personal project linked in CV

[Auto-screened by AI — final decision rests with recruiter]
```

### Evaluation Output Type

- **D-09:** New `EvaluationResult` type in `src/agent/types.ts` with fields: `applicationId`, `applicantId`, `outcome` (`'pass' | 'fail' | 'needsReview'`), `required` (array of `{ label, met, rationale }`), `optional` (array of `{ label, met, rationale }`), `comment` (the formatted recruiter comment string), `timestamp`.

- **D-10:** `EvaluationResult` is logged directly to stdout for soft-evaluated candidates (replacing the Phase 2 `CandidateDecision` placeholder for the `pass` branch). `CandidateDecision` continues to be used for all other paths: hard-rule fails, PDF needsReview, errors, unhandled exceptions.

- **D-11:** `EvaluationResult` is also the object that Phase 4 will consume to make BambooHR write calls (`BAMB-02` stage transition + `BAMB-03` comment write). Its `comment` field is the ready-to-post string; `outcome` drives the target stage.

### Agent Architecture

- **Claude's discretion:** The OpenAI Agents SDK is used for the run/maxTurns loop (`Runner.run()`, `maxTurns ≤ 5` per SAFE-02). No tools are needed — all candidate data is already present in `CandidateContext`. The agent receives the full context in the system + user prompt and returns a structured output matching the `EvaluationResult` schema. This satisfies SAFE-02's maxTurns cap without tools (single-turn structured-output call never exceeds 1 turn). The SDK is used for its agent run abstraction rather than tool-use orchestration.
- File layout for Phase 3 code: `src/agent/` directory (types.ts, evaluator.ts, prompt.ts — Claude decides internal split).
- Config schema extension: `softRules` added as an optional top-level key in `configSchema` (optional so existing configs without soft rules remain valid; a config with soft rules requires at least one `required` entry).

<specifics>
## Specific Ideas

- The `EvaluationResult` structured output schema is also the GPT-4o response schema — the same Zod schema (or TypeScript type) should drive both the SDK's structured output call and the `EvaluationResult` interface. This avoids two separate type definitions that can drift.
- Comment is pre-formatted by GPT-4o as a ready-to-post string (not assembled from parts by code). GPT-4o receives the structured per-criterion data AND generates the comment in the same response — both in the structured output schema.
- The `[Auto-screened by AI — final decision rests with recruiter]` footer is hardcoded in the GPT-4o system prompt (not configurable). This is a fixed audit disclosure, not a user-visible config option.
- If `softRules` is absent from config.yaml (or has no `required` entries), Phase 3 should skip GPT-4o entirely and log the candidate as `pass` with empty required/optional arrays and comment `'No soft rules configured'`. This keeps the pipeline functional during testing and maintains backward compatibility with Phase 1/2 configs.
